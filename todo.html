Game Mechanics

The game should have the following mechanics implemented in a modular way, so that the rest of development is scalable:

  1. A functional dialogue system which is node based. The dialogue system can have fields such as "speaker name", "dialogue line", "speaker animation", "speaker expression", "speaker name color", "recipient animation", "recipient expression" and a field to invoke events like the ones that will be mentioned in the next entry.
  2. A task system with at least one type of task available. Examples are: a collectable task, an escort task, a service task, etc.. Events are actions where the task is discovered or recieved, completed or failed.
  3. A scene to test the features. (like a classroom, natural scenery, etc.)

Story Content

At this time I should have devised some of the characters as well as their relationships with each other. Note: relationships does not need to mean romantic relationships. Idealy a story should have been developed along side it but I think it is more important to have complete characters which are not just there for very obvious story-specific reasons. A story revolves around people, their actions and their view of the world, not people existing purely for the purpose of a story.

Visual Aesthetics

The games 3D models will look like a mixture of the original games anime design but will also adhere to the realism of the assets so as to not contrast with them too much. Colors should be not too saturated nor should it have an ugly grey-filter or sepia-filter effect on them like a lot of games out there do.

Models will also not be color-swapped copies of eachother (like how it looked like in the original game due to using only one base asset for all characters) and will also have more realistic features (such as dark undereyes for some, birthmarks, scars, etc.)

Bug Hunting

As the title implies, this phase will revolve around testing the features in their entirety (as opposed to testing specific features right after development) to see what works and what doesn't. This time, community input is essential to ensure a successful reimagination of the original game.

Accessibility

With input from the community, we can make some changes to ensure people with e.g. visual impairments like colorblindness, severe myopia and more can enjoy not only the game, but also easily navigate the website. Same thing applies to other impairments like motor issues (so quick time events, if they are implemented, should remain optional) Of course not all can be adjusted for each person, as this is still developed by me (some random teenage girl).
